# Rumina 鬼教官 — 音声処理API接続仕様書 v1.0

> MVP（モックUI）を実際の音声解析に接続するための実装仕様。
> 主軸エンジンは **OpenAI Whisper API**。ただし `TranscriptionProvider` インターフェースで
> Google STT / AssemblyAI / Deepgram に無改修で差し替え可能な設計とする。

---

## 0. なぜ Whisper 主軸か（推奨理由）

| 観点 | 評価 | 補足 |
|---|---|---|
| 日本語精度 | ◎ | 訪販の口語・ノイズ環境でも実用水準。方言もそこそこ拾う。 |
| 導入速度 | ◎ | REST 1本。話者分離を最初は捨てれば即日繋がる。 |
| コスト | ○ | $0.006/分。7時間で約 $2.5/人日（後述）。 |
| 弱点 | △ | **話者分離(diarization)を持たない**・タイムスタンプが粗い区間がある。 |

→ **戦略**：まず Whisper で「文字起こし＋粗タイムスタンプ」を取り、話者分離・区間検出は
**別レイヤ（VAD＋GPS＋ルール）** で補う。精度が要件を超えたら該当区間だけ AssemblyAy/Deepgram に回す。

## 1. 全体パイプライン

```
録音(最大7h)
  │  ① Upload         … 署名付きURLで直PUT。7hを一括で送らない
  ▼
  ② Preprocess        … 16kHz/mono正規化 → 無音(VAD)で 10〜15分チャンクに分割
  ▼
  ③ Transcribe        … チャンク単位でWhisperへ並列投入 → タイムスタンプ付きテキスト
  ▼
  ④ Stitch            … チャンクを時間軸で連結・重複(overlap)除去
  ▼
  ⑤ Segment (ピンポン) … 無音ギャップ＋GPS移動＋チャイム音で「訪問1件」に切り分け
  ▼
  ⑥ Extract (KPI)     … 各PingEvent/1日サマリを機械算出（在宅/会話/切り返し/冒頭質問…）
  ▼
  ⑦ Diagnose          … benchmark差分 → 弱点/注意を自動判定（diagnose.js の本番版）
  ▼
  ⑧ Coach             … LLMで鬼教官コメント＋明日の改善アクション生成
```

処理は**全て非同期ジョブ**。UIは `status` をポーリング（または webhook）で進捗表示する。

## 2. APIルート（既存の想定を踏襲）

| Method | Path | 役割 |
|---|---|---|
| POST | `/api/audio/upload` | 署名付きURL発行＋`AudioSession`を`queued`で作成 |
| POST | `/api/audio/transcribe` | ②③④を起動（内部的にはupload完了webhookから自動でも可） |
| POST | `/api/audio/analyze` | ⑤⑥⑦⑧を起動 |
| GET | `/api/sessions/:id` | ステータス＋進捗（chunk done/total, stage） |
| GET | `/api/reports/:id` | `AnalysisResult`＋`PingEvent[]`＋診断＋講評 |
| GET | `/api/sales-reps` | 一覧＋当日サマリ |

### ステータス機械（`AudioSession.status`）
`queued → uploading → transcribing → segmenting → analyzing → done`（失敗時 `failed`、部分成功 `partial`）
UIの解析中画面5ステージはこの遷移に1:1で対応済み。

## 3. ② 前処理・チャンク分割

- **正規化**: 16kHz / mono / ラウドネス正規化（-16 LUFS目安）。ffmpegで実施。
- **分割単位**: 10〜15分。境界は**必ず無音(VAD)位置で切る**。会話の途中で切らない。
- **overlap**: 各チャンク末尾に 3〜5秒の重複を持たせ、④で重複テキストをdedup。境界の欠落を防ぐ。
- **VAD**: `webrtcvad` か `silero-vad`。無音閾値・最小無音長(例: 700ms)をチューニング可能に。

```jsonc
// chunk メタ
{ "index": 12, "startSec": 8640, "endSec": 9540, "hasSpeech": true, "silenceRatio": 0.18 }
```

## 4. ③ 文字起こし（TranscriptionProvider）

差し替え可能にするための共通インターフェース。**この境界だけ守れば全エンジン互換**。

```ts
interface TranscriptSegment {
  startSec: number;         // チャンク基準 → ④で日内絶対秒に補正
  endSec: number;
  text: string;
  speaker?: string | null;  // Whisperは基本 null。diarization併用時のみ埋まる
  confidence?: number;      // 低信頼セグメントは⑥で減点/除外
}
interface TranscriptionProvider {
  name: 'whisper' | 'google' | 'assemblyai' | 'deepgram';
  transcribe(chunk: AudioChunk, opts: { lang: 'ja'; prompt?: string }): Promise<TranscriptSegment[]>;
}
```

### Whisper 呼び出し（推奨パラメータ）
- `model: whisper-1`、`language: "ja"`、`response_format: "verbose_json"`（segmentsとタイムスタンプ取得）、`temperature: 0`。
- **`prompt` にドメイン語彙をバイアス**として渡す（誤認対策・後述）:
  例）`"電気代 明細 無料診断 太陽光 蓄電池 検針票 kWh 東京電力 オール電化 インターホン"`
- タイムアウト・リトライ（指数バックオフ、最大3回）、**チャンク単位の冪等キー**でリトライ時の二重課金を防ぐ。

### 出力（時間情報付き・UI互換フォーマット）
```
[00:12:10] 営業：こんにちは、電気代の無料診断で回っています。
[00:12:15] お客様：大丈夫です。
```
`who`（営業/お客様）は Whisper では出ないため **⑤の話者推定で後付け**する。

## 5. ⑤ ピンポン自動分割（このアプリの心臓部）

Whisperは「誰が」「どの家か」を知らない。**音声以外の信号を主軸に切る。**

| 手がかり | 使い方 | 信頼度 |
|---|---|---|
| **GPS移動** | 30m以上の移動＝別の家。停止クラスタ＝1訪問 | ★★★（最優先） |
| 無音ギャップ | 長い無音（例>90s）＝訪問間の移動 | ★★ |
| チャイム/インターホン音 | ピンポン検出 → 訪問開始マーカ | ★★（要音響モデル） |
| 加速度/歩行センサ | 歩行中＝移動、静止＝接客 | ★★（スマホ併用時） |
| 発話ターン | 「はい」「こんにちは」等の開始定型句 | ★ |

→ **GPS停止クラスタを主、音響/無音を従**にして `PingEvent` を確定。GPSが無い運用では
無音ギャップ＋チャイム音のみで切るが、精度が落ちる旨をUIに明示する（正直表示）。

### PingEvent 判定ロジック（各1件について）
```
result: apo | prospect | talk | reject | nohome | intercom
customerReaction: 在宅 / 不在 / インターホンのみ / 対面
```
- **在宅/不在**: 訪問区間にお客様側発話があれば在宅。チャイム後15秒以内に応答無し＝不在。
- **インターホンのみ / 対面**: 音の反響・音量・背景ノイズ差で推定（対面はレンジ広い）。確度低い時は「不明」に倒す。
- **会話発生**: お客様発話ターン ≥ 1 かつ会話長 ≥ 閾値(例10s)。
- **切り返し回数**: 「断り発話 → 直後の営業発話」ペアの回数。断り検出は下記分類器。
- **アポ化/見込み**: 終盤に日時語・同意語（「じゃあ」「見てみる」「◯日」）＋LLM判定。

## 6. ⑥ KPI機械抽出（各指標の算出元）

| KPI | 算出方法 |
|---|---|
| 総ピンポン数 | 確定した`PingEvent`件数 |
| 在宅反応数/率 | reaction∈{在宅,対面,インターホンのみ} の件数 / 総数 |
| 会話発生数/率 | 会話発生フラグ true の件数 / 総数 |
| 平均会話秒数 | 会話区間長の平均 |
| **冒頭質問率** | 各対面の**最初10秒**に「？」or疑問詞orキーワード(電気代/明細/診断)を含む率 |
| **切り返し回数** | §5の断り→営業ペア数の平均 |
| 断り文句TOP5 | 断り区間テキストを分類器/LLMで正規化 → 集計 |
| アポ数/率 | result=apo / 総数 |
| 無音時間 | VADの無音合計 |
| **サボり疑い時間** | GPS停止でも訪問でもない空白 or 稼働帯の連続無活動（**§9の誤判定注意あり**） |
| 移動時間 | GPS移動区間の合計 |

**断り文句分類**: まずキーワード辞書（「間に合ってる」「忙しい」「持ち家じゃない」「主人が」…）で
一次分類、外れたものだけ LLM に `label ∈ 既知カテゴリ|other` で分類させる（コスト最適化）。

## 7. ⑦ 自動診断エンジン（何がダメ／何に気をつけるか）

**MVPの `diagnose.js` がそのまま本番仕様**。実データの `AnalysisResult` を入れれば同じ判定が走る。

### 判定は「閾値ルール＋テンプレ生成」（LLM非依存＝安定・低コスト・説明可能）
- 各KPIを**トップ営業比のratio**で評価し、`crit`/`warn`の2段階閾値で弱点化。
- 弱点1件ごとに `{finding（数字）, cause（原因仮説）, fix（処方）}` を自動生成。
- 注意(caution)は「リスク⚠」と「伸びしろ💡」の2種。時間帯・断り偏り・データ品質から導出。

### 現行の閾値表（`diagnose.js` の `TH`／調整可能）
| 指標 | warn (トップ比) | crit (トップ比) |
|---|---|---|
| 行動量 | <90% | <80% |
| 在宅反応率 | <90% | <75% |
| 会話時間 | <90% | <70% |
| 切り返し | <80% | <50% |
| 冒頭質問率 | <90% | <60% |
| アポ率 | <100% | <60% |

### 注意(caution)の自動生成トリガ
- **空白時間** ≥20分 → ⚠（※移動/昼食/電波断の誤判定可能性を必ず併記＝正直表示）
- **無音** ≥60分 → ⚠データ信頼性（録音が拾えていない区間の疑い→GPS突合）
- **断り集中** TOP1が全体の25%以上 → ⚠その一言への切り返し弱い
- **夕方在宅濃い** のに訪問薄い → 💡伸びしろ
- **昼の失速**（12-13時が他帯の6割未満）→ 💡

> 本番では benchmark を「固定仮値」から**チーム実データのトップN平均に自動更新**する
> （日次バッチ）。これで「トップ営業の型を全員にコピー」が生きたデータで回る。

## 8. ⑧ 鬼教官コメント／改善アクション（LLM）

- 入力：`AnalysisResult` + `diagnose()`結果 + 代表`PingEvent`抜粋（悪い接客・良い接客の実文字起こし）。
- 出力スキーマ（JSON強制）:
```jsonc
{ "blocks":[{ "tone":"harsh|good|close", "title":"", "text":"" }],
  "nextActions":[{ "icon":"", "text":"", "metric":"" }] }
```
- **システムプロンプト方針**（世界観固定）:
  「甘やかすな／人格否定は禁止／数字・行動・会話だけを詰めろ／必ず具体例を出せ／
  最後は"ここを直せば勝てる"で締めろ」。実発話を引用して指摘の説得力を上げる。
- 推奨モデル：Claude（例 `claude-sonnet-5`）。温度低め。診断が事実、LLMは表現に専念。

## 9. ⚠ 実装上の注意・落とし穴（＝何を気をつけるべきか）

このアプリは**人事評価に直結し、他人の声を録音する**。技術以上に運用・法務が地雷。

### A. 文字起こしの誤認
- 「電気」→「天気」、「明細」→「名刺」等の同音誤認。→ **domain prompt bias**＋固有語辞書で後補正。
- confidence低いセグメントを冒頭質問判定に使うと数字が暴れる。→ 低信頼は減点/除外。

### B. 話者取り違え（最大リスク）
- 営業とお客様を逆に割り当てると、**切り返し・冒頭質問・会話時間が全部壊れる**。
- Whisper単体は話者を出さない。→ diarization併用 or「営業＝常時装着マイクで音量大」等の
  物理前提を置く。取り違え疑い時はその区間をKPIから除外し、UIに「要確認」を出す。

### C. サボり判定の誤検知（労務・信頼問題）
- 空白＝移動/昼食/トイレ/電波断/エレベータの可能性。**単独で"サボり"と断定しない**。
- 必ず **GPS・明細・チャイムログと突合**してから提示。UIも「疑い」表記＋裏取り導線を固定。
- 診断エンジンは既にこの但し書きを自動付与済み（§7 caution A）。

### D. 個人情報・録音同意（要法務確認）
- **お客様の声**が入る。地域の録音・保存要件、保持期間、マスキング（顧客氏名/住所の伏字）を設計。
- 文字起こしはお客様発話を**要点のみ保存**し、生音声は短期で破棄する運用を推奨。

### E. 労基・監視ツール性
- 常時録音＋サボり検知は**従業員監視**に該当し得る。就業規則・同意・利用目的の明示が前提。
- 「評価」ではなく「教育・改善」目的を明確にし、詰めの表現がハラスメントにならない設計に。

### F. コスト暴発
- Whisper $0.006/分。7h=420分 → **約$2.5/人日**。50人×22日 ≈ **$2,750/月**（文字起こしのみ）。
- LLM講評・断り分類のトークンが上乗せ。→ 断り分類は辞書一次でLLM呼び出しを削る、
  無音チャンクは③をスキップ、講評は1日1回に限定。

### G. 長時間ジョブの信頼性
- 7hを1リクエストにしない（タイムアウト/失敗時の全損）。チャンク単位で**冪等・部分再実行**。
- チャンク一部失敗でも `partial` で残りを出す。二重課金防止に冪等キー必須。

### H. 静かな失敗
- VAD閾値ミスで会話を無音と誤判定→数字が実態より低く出て、無実の営業を詰める事故。
- **無音率が異常に高いセッションは自動でフラグ**し、講評前に人間レビューへ回す。

## 11. 段階導入プラン

1. **Phase 1（繋ぐ）**: upload→Whisper→文字起こし表示まで。ピンポン分割は無音ギャップのみ。
2. **Phase 2（切る）**: GPSログ連携でピンポン分割の精度確保。KPI機械抽出。
3. **Phase 3（詰める）**: 診断エンジン本番化＋benchmark自動更新＋LLM講評。
4. **Phase 4（守る）**: 録音同意フロー・マスキング・監視ログ・品質フラグの人間レビュー導線。

> Phase 1 は既存UIに `provider=whisper` を挿すだけで動く。まずここまでを最短で繋ぐことを推奨。
```
